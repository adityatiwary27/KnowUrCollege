API endpoints & examples
- Register user (hashes password):
  - POST `/api/users/register`
  - Body example:
    ```json
    {"username":"alice","email":"alice@example.com","password":"secret"}
    ```
- Login (get JWT):
  - POST `/api/auth/login`
  - Body example:
    ```json
    {"username":"alice","password":"secret"}
    ```
  - Response:
    ```json
    {"token":"<JWT_TOKEN>"}
    ```

- Create post (JSON):
  - POST `/api/posts`
  - Body example:
    ```json
    {"user":{"id":1},"content":"Hello world"}
    ```

- Create post with image (server uploads to Cloudinary)
  - POST `/api/posts` as multipart/form-data:
    - Part `post` = JSON string (application/json)
    - Part `image` = file
  - Curl example:
    ```bash
    curl -X POST \
      -H "Accept: application/json" \
      -F 'post={"user":{"id":1},"content":"Hello with image"};type=application/json' \
      -F "image=@/path/to/photo.jpg" \
      http://localhost:8081/api/posts
    ```
  - Response includes `imageUrl` (Cloudinary secure URL) and `imagePublicId`.

- Direct signed upload flow (recommended for large files)
  1. Backend: request signature
     - POST `/api/cloudinary/sign` with optional body params (e.g., `{"folder":"user_uploads"}`)
     - Response: `{"api_key":"...","cloud_name":"...","timestamp":123456,"signature":"..."}`

  2. Frontend: upload directly to Cloudinary:
     - Upload URL: `https://api.cloudinary.com/v1_1/<cloud_name>/auto/upload`
     - Form fields to include: `file`, `api_key`, `timestamp`, `signature`, plus any signed params.
     - After upload Cloudinary returns `secure_url` and `public_id`. Then call your backend to create a Post referencing those values, or you can send a webhook.

  - Example server signing request (curl):
    ```bash
    curl -X POST -H "Content-Type: application/json" -d '{"folder":"user_uploads"}' http://localhost:8081/api/cloudinary/sign
    ```

  - Example minimal client JS (browser) to request signature and upload file:
    ```javascript
    async function signAndUpload(file) {
      // 1) get signature from backend
      const signResp = await fetch('/api/cloudinary/sign', {
        method: 'POST',
        headers: {'Content-Type': 'application/json'},
        body: JSON.stringify({folder: 'user_uploads'})
      });
      const sign = await signResp.json(); // { api_key, cloud_name, timestamp, signature }

      // 2) upload to Cloudinary directly
      const form = new FormData();
      form.append('file', file);
      form.append('api_key', sign.api_key);
      form.append('timestamp', sign.timestamp);
      form.append('signature', sign.signature);
      // include folder if used:
      form.append('folder', 'user_uploads');

      const uploadUrl = `https://api.cloudinary.com/v1_1/${sign.cloud_name}/auto/upload`;
      const uploadResp = await fetch(uploadUrl, { method: 'POST', body: form });
      const uploadJson = await uploadResp.json();
      // uploadJson.secure_url and uploadJson.public_id are returned
      return uploadJson;
    }
    ```

- Delete post image (admin)
  - DELETE `/api/admin/posts/{id}/image`
  - Requires Authorization header: `Authorization: Bearer <JWT_TOKEN>`
  - Example:
    ```bash
    curl -X DELETE -H "Authorization: Bearer <JWT_TOKEN>" http://localhost:8081/api/admin/posts/<POST_ID>/image
    ```

How Cloudinary is wired in code (quick pointers)
- `CloudinaryService` constructor reads `CLOUDINARY_URL` env var and constructs `new Cloudinary(url)`. Methods:
  - `upload(MultipartFile)` → returns map with `url` and `public_id`.
  - `destroyByPublicId(String)` → deletes resource.
- `PostService.createWithImage(...)` calls `cloudinaryService.upload(...)` if `image` present and stores `imageUrl` and `imagePublicId` in `Post`.
- `CloudinaryController` builds signature for signed direct uploads using `CLOUDINARY_URL` secret.

Security notes & cautions
- I used `hibernate.ddl-auto=update` for quick verification; migrate to Flyway/Liquibase for production.
- `JWT_SECRET` should be set in production (not randomly generated) and stored in secrets manager.
- Do not commit `CLOUDINARY_URL` or DB credentials.
- The current security config permits GETs publicly and protects mutation endpoints; adjust roles/authorities for production.
